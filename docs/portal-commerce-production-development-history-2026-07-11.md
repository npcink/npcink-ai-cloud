# Portal 商业化与生产收口开发历史

状态：截至 2026-07-11 的阶段总结

范围：Portal 商业化、后台信息架构、身份授权、邮件与密钥、订单生命周期、生产发布与历史兼容清理

## 1. 文档目的

本文把 2026 年 6 月底至 7 月 11 日围绕 Portal 上线准备的讨论、实现、故障和生产验证归并为一条可追溯主线。它不是新的产品规范，也不替代各专项契约；它用于回答三个问题：

1. 当前系统已经做到了什么；
2. 为什么采用现在的模型，而没有继续保留历史双轨设计；
3. 后续开发应遵守哪些边界和验证方法。

专项规则仍以本文末尾列出的契约、验收记录和发布文档为准。

## 2. 阶段结论

当前阶段已经形成一条 PC 优先、最小但完整的商业闭环：

- 套餐顺序固定为 `Free < Plus < Pro < Agency`；
- Plus、Pro 支持自助购买，Agency 保留账户级运营审批和报价；
- Plus、Pro、Agency 均可作为 14 天试用目标，但同一客户只共享一次付费套餐试用资格；
- 积分包可在后台维护并由 Portal 用户购买，有效期统一为 365 天；
- 人民币是当前后台和 Portal 的唯一计价口径；
- 每个账户最多同时存在 5 笔未过期未支付套餐订单，订单 30 分钟未付款自动失效；
- Portal 登录和站点访问只认账户成员关系，不再依赖 `site_user_grants` 双重授权；
- 服务配置和 Provider Connection 密文使用独立、稳定的服务配置密钥；
- 生产发布、备份、恢复、密钥迁移、订单过期和限额均完成过真实环境验证。

支付宝支付和 QQ 登录目前保持禁用，密文配置可以保留，待提供方配置完成后再执行真实回调验收。禁用状态不应阻塞本地和生产环境中的模拟订单生命周期验证。

## 3. 不可突破的 Cloud 边界

这些商业和运营能力属于 Cloud 的托管服务职责，但没有改变产品边界：

- Cloud 可以持有账户、套餐、订阅、支付订单、额度、用量、服务健康和运行证据；
- Cloud 可以提供只读运行详情和明确的客户操作入口；
- WordPress 仍是本地 ability、workflow、prompt、preset 和最终写入审批的控制面；
- Cloud 不得成为第二个 WordPress 控制面、第二个 ability registry 或第二个 workflow registry；
- Cloud 产生的内容、诊断和建议仍应保持 `suggestion_only`，不能绕过本地审核和审计。

因此，本阶段没有把套餐购买扩展成 WordPress 功能开关，也没有让 Portal 接管 WordPress 的治理真相。

## 4. 商业模型的收口过程

### 4.1 套餐定位

Plus 被明确放在 Free 和 Pro 之间，用于承接免费用户的首次付费升级。当前套餐层级和购买方式为：

| 套餐 | 定位 | 当前购买方式 | 当前周期价格 |
| --- | --- | --- | --- |
| Free | 免费入口和基础体验 | 注册后获得 | ¥0 |
| Plus | 入门付费套餐 | Portal 自助购买 | ¥15 / 30 天 |
| Pro | 主要个人或小团队套餐 | Portal 自助购买 | ¥29 / 30 天 |
| Agency | 多站点和更高资源上限 | 账户级审批和报价 | 不使用匿名固定价 |

套餐升级遵循向上移动：Free 可以升级到 Plus、Pro 或 Agency，Plus 可以升级到 Pro 或 Agency，Pro 可以升级到 Agency。降级、退款和立即生效策略必须由订单与订阅状态明确表达，不能依赖前端隐藏按钮来维持约束。

### 4.2 14 天试用

Plus、Pro、Agency 都可以作为试用目标，但试用资格属于客户，而不属于套餐：

- 每个客户只有一次 14 天付费套餐试用；
- 试用期间向上切换套餐不重置结束时间；
- 切换套餐不重复发放试用积分；
- 试用结束后必须通过正式订单进入付费订阅；
- 试用资格、订阅状态和额度发放必须能够被后台解释和审计。

这样可以避免用户在三个套餐间循环领取 42 天试用，也避免把促销规则散落在 UI 中。

### 4.3 积分包

积分包是订阅额度之外的补充，不替代套餐：

- 后台通过积分包目录维护可售规格；
- Portal 用户可为所属账户购买积分包；
- 当前有效期固定为 365 天，从成功发放时开始计算；
- 购买、支付、发放和到期必须落入账本或可审计证据；
- 消耗顺序和过期处理以积分扣费契约为准，不能由页面自行计算。

一年有效期兼顾了用户预期和会计可解释性。永久有效会形成长期负债，过短有效期则容易造成售后争议。

## 5. 套餐与后台信息架构

### 5.1 套餐目录

后台套餐目录承担“当前有效商业定义”的展示和管理，不再把套餐状态混入服务状态页。PC 端以一行四个套餐的长列表展示 Free、Plus、Pro、Agency，便于横向比较。

“套餐初始化”只用于缺少标准套餐时的种子或引导创建，不是日常运营入口。当标准套餐均已存在并发布时，该入口可以不显示；运营人员应通过套餐目录管理现有套餐，而不是重复初始化。

### 5.2 移除重复入口

本阶段对后台导航进行了职责收口：

- 移除服务状态工作区里的“套餐概览”Tab，套餐目录保持独立一级页面；
- “服务风险”不再作为独立一级入口，改为服务状态页中的“打开订阅风险”二级动作；
- `/admin/accounts` 是客户账户主目录；
- `/admin/portal-users` 仅保留自注册 Portal 用户的辅助管理语义，不作为第二套客户真相；
- 后台金额统一使用人民币和 `¥`，不混用美元展示。

### 5.3 Portal 工作区职责

Portal 页面按用户任务分工：

- 套餐：账户拥有什么；
- 用量：账户消耗了什么；
- 账单：账户订购和支付了什么；
- 站点：账户连接了哪些 WordPress 站点；
- 支持：当前问题如何进入可处理流程。

普通 Portal Shell 不再使用全局站点选择器制造隐式上下文。账户是套餐、用量和账单的主边界，站点只在确实需要站点上下文的页面中选择。

## 6. 身份与授权模型

产品身份当前只保留两种：

- `platform_admin`：平台管理员；
- `user`：普通 Portal 用户。

运营人员角色明确延期，当前不通过新增角色、隐藏权限组合或文案别名提前引入。

`AccountUserMembership` 已成为 Portal 授权的唯一真相。历史 `site_user_grants` 模型经历了以下退出过程：

1. 补齐账户成员关系并停止依赖站点授权回退；
2. 将列表、详情和插件绑定统一到账户成员判断；
3. 停止写入旧站点授权；
4. 通过数据库迁移删除 `site_user_grants` 表；
5. 在生产数据库和恢复副本中确认旧表不存在。

这解决了“列表看得到站点，点进去却 403”的双重权限真相问题。插件绑定流程同时保留账户上下文和完整重定向参数。开发环境应统一使用 `localhost` 或 `127.0.0.1`，避免 Cookie 主机不一致导致登录态丢失。

## 7. 邮件验证码与 SMTP

历史问题表现为第一次邮件发送成功，第二次或第三次看似收不到。排查后确认不能把所有情况简单归因于 SMTP：请求限流、验证码意图复用、重发入口不清、倒计时缺失和提供方投递延迟共同造成了“没有发送”的感知。

当前处理原则为：

- 注册和登录使用明确的验证码意图；
- 同一邮箱短窗口限额从 3 次调整为 5 次，同时保留反滥用保护；
- 重发按钮显示倒计时和禁用状态；
- 错误区分限流、验证码错误、验证码过期和邮件投递失败；
- 接受标准有效邮箱格式，不通过屏蔽特定邮箱域名代替反滥用；
- API 和 UI 不泄露 SMTP 凭证或提供方内部错误。

生产 SMTP 已完成连续三封真实邮件投递验证。后续调试应优先使用受控测试收件箱、缩短人工操作链路和读取明确错误码，而不是反复向真实用户邮箱发送邮件。

## 8. 服务配置密钥

### 8.1 问题根因

历史 SMTP 密码密文无法解开的根因是：密文曾使用会随运行环境变化的会话或令牌派生密钥。密文仍在数据库中，但运行时密钥身份已经改变，因此保存配置时读取旧密文会触发 500。

### 8.2 最终设计

服务配置采用专用且稳定的 `NPCINK_CLOUD_SERVICE_SETTINGS_SECRET`：

- SMTP 等服务设置使用该密钥加密；
- Provider Connection 使用同一根密钥但保持独立 Fernet purpose；
- 不再回退到管理员会话密钥、Portal JWT 或内部服务令牌；
- 生产环境必须显式配置并长期保存该密钥；
- 密文无法解开时明确失败，由管理员重新配置或执行受控迁移，不能静默清空。

生产迁移先备份数据库，再验证旧密钥可解密全部 8 条 Provider Connection，生成不含明文的旧新密文映射，事务性重加密后部署，并确认全部记录能被新密钥读取。

这一设计没有增加不必要的业务复杂度。它把“密钥身份”从偶然的运行时状态提升为持久数据契约，减少了隐式回退和不可解释行为。安全关键在于密钥单独保管、文件权限、轮换流程和备份，而不是减少一个配置项。

## 9. 支付订单生命周期

### 9.1 未支付订单策略

原行为是单笔未支付订单阻塞后续下单，导致一笔 ¥15 测试订单长期占用购买入口。当前策略调整为：

- 每个账户最多保留 5 笔未过期的未支付套餐订单；
- 第 6 笔返回 `service.subscription_order_pending_limit`；
- 未付款订单 30 分钟后自动失效；
- 托管运维任务每分钟执行一次 `payment_order_expiration`；
- 用户取消订单时必须确认支付提供方关单成功，失败时保持原状态；
- 任一订单支付成功后关闭同账户的其他待支付同类订单；
- 取消或自动过期释放名额，用户可以重新下单。

允许少量并存订单解决了误关页面、切换套餐和支付渠道失败后的恢复问题；5 笔上限和 30 分钟期限则限制了垃圾订单、库存式占位和后台噪声。

### 9.2 失败案例与结论

生产验收时发现支付宝和 QQ 实际处于 `enabled=True/ready`，与“暂缓真实配置”的产品决定不一致。真实支付宝关单返回无效或非 JSON 响应，系统按照 fail-closed 规则拒绝把订单标记为已取消，因此测试订单仍占用名额。

处理方式不是绕过关单保护，而是：

1. 通过正式过期逻辑清理已超过 30 分钟的测试订单；
2. 禁用支付宝和 QQ，但保留加密凭证；
3. 在支付宝禁用状态下使用模拟网关验证取消和重新下单；
4. 将真实支付宝回调和 QQ 登录留到提供方配置完成后单独验收。

这次事件确认了两个原则：配置的 `enabled` 状态必须反映真实可用性；涉及资金状态转换时，外部确认失败必须关闭式失败，不能为了界面顺畅伪造成功。

## 10. 历史兼容与未使用能力清理

上线前完成了以下运行时兼容清理：

- 删除 Portal 旧 `site_user_grants` 回退和表；
- 修复 PR #146 暴露的 Mypy 错误并恢复 CI；
- 删除旧环境变量别名和旧 QQ 回调路径；
- 强制使用服务配置专用密钥并删除旧密钥回退；
- 移除最后两项 Mypy 债务豁免；
- 拒绝请求级 runtime callback 覆盖，只接受注册站点的终端回调；
- 删除 Provider Connection 旧回退密钥和无用旧 Cookie；
- 删除没有业务调用方且与购买能力冲突的 Feature Flags facade。

能力清理遵循“先查真实消费者，再决定删除”的方法：

- Operations Advisor 从首发导航隐藏，但保留直达页和 API；
- 音频生成和音频资产仍被 Toolbox 与 Cloud Addon 使用，继续保留；
- Site Knowledge 评论仍有真实消费者，保持默认关闭和显式启用；
- Agent Feedback 与模型参考情报作为只读运行详情保留。

历史迁移、`docs/legacy-contracts` 和负向拒绝测试没有删除。它们不参与运行时回退，是数据库升级、边界审计和防止旧行为复活的证据。

## 11. 生产发布与恢复演练

生产发布采用固定分支和证据链：

- `master` 是开发集成分支；
- `production` 是生产发布源；
- 从 CI 绿色的 `master` 创建生产提升 PR；
- PR 必须包含 `Approved for production validation by operator.`；
- 部署前确认范围、数据库备份和回滚路径；
- 部署后执行 API、Frontend、数据库迁移和生产 Smoke；
- 对有状态功能执行小规模真实数据验证，并清理测试数据；
- 把备份、校验和、运行结果和异常处理记录到发布证据。

本阶段完成了：

- 服务器由 2 GB 升级到 4 GB 后的数据库恢复演练；
- 订阅迁移后的恢复副本核验；
- 服务配置密钥和 Provider Connection 重加密；
- 未支付订单自动过期与 5 笔上限的生产验证；
- 创建 5 笔成功、第 6 笔受限、取消后可重新创建的完整验证；
- 测试订单清理，最终生产待支付套餐订单数恢复为 0。

开发和测试专用服务不部署到生产。生产只保留 API、前端、运行 Worker、回调 Worker、运维 Worker、PostgreSQL、Redis、反向代理以及实际使用的可观测组件。

## 12. 开发方法沉淀

### 12.1 先确定真相，再做界面

套餐、订单、权限和密钥问题都证明：如果底层存在两个真相，调整页面只能掩盖问题。应先明确唯一权威，再让 API 和 UI 投影该权威。

当前唯一权威包括：

- Portal 授权：账户成员关系；
- 服务和 Provider 密文：服务配置专用密钥；
- 付费激活：支付回调和可验证的支付状态；
- 套餐、用量与账单边界：账户；
- WordPress 最终写入和本地治理：WordPress/Core。

### 12.2 资金、凭证和授权转换关闭式失败

支付关单失败不能伪造取消，密文解不开不能静默清空，缺少账户成员关系不能从旧站点授权猜测权限。明确失败会增加一次人工处理，但能避免不可逆的数据和安全问题。

### 12.3 业务状态与提供方适配分离

订单的 30 分钟期限、5 笔上限、兄弟订单关闭和幂等规则属于确定性业务逻辑；支付宝、模拟网关和未来支付渠道属于适配器。业务规则不应依赖某一提供方响应格式。

### 12.4 删除可执行兼容，保留历史证据

旧回退、旧别名和双写路径会持续扩大测试矩阵，应在迁移完成后删除。迁移文件、历史契约和拒绝测试则用于解释数据库与边界演化，应继续保留。

### 12.5 生产验证是功能交付的一部分

本地测试无法发现启用状态漂移、真实支付响应、旧密文和运维定时任务问题。完整交付顺序应为：

1. 小范围实现和针对性测试；
2. `master` CI；
3. `production` 提升 PR；
4. 生产环境批准和备份；
5. 部署 Smoke；
6. 受控真实数据验证；
7. 清理测试数据并记录证据。

### 12.6 用简化获得性能

当前阶段的性能优先级不是增加更多服务，而是减少双写、隐式回退、无界订单和重复页面。PC 端先做完整主流程，移动端延期；生产只运行实际需要的服务；所有定时清理都有边界和固定节奏。

### 12.7 使用干净工作树提交独立变更

发布、文档和修复应使用范围清晰的分支和提交。主工作树存在并行改动时，使用独立 worktree，逐文件暂存并核对 staged diff，避免把无关代码带入 PR。

| 支付宝真实付款、回调、退款 | 延后 | 当前禁用，待正式配置 |
| QQ 真实登录 | 延后 | 当前禁用，待正式配置 |
| 运营人员角色 | 延后 | 当前只保留管理员和普通用户 |
| 移动端专项适配 | 延后 | 当前 PC 优先 |

## 14. 下一阶段触发条件

当前阶段可以结束，不需要继续扩张后台页面。只有在以下条件满足时再开启对应工作：

- 支付宝商户参数可用后，执行小额支付、异步回调、额度发放、关单和退款端到端验收；
- QQ 应用和回调域名可用后，执行真实登录和账户绑定验收；
- 出现真实运营团队后，再设计运营角色和最小权限矩阵；
- PC 主流程积累真实用户反馈后，再做移动端布局和交互适配；
- 真实负载数据表明 4 GB 或当前 Worker 配置不足后，再基于指标扩容，而不是预先堆叠服务。
